## 一、项目目标速览

### 1.1 我们要做什么？

想象一下这个系统就像一个**"资讯翻译官"**：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   微信公众号文章                    最终结果                      │
│   （散落各处、格式混乱）     →     （整理好的资讯包）              │
│                                                                 │
│   比如：                               比如：                     │
│   • 某公众号发了篇文章           • 每天早上9点收到邮件             │
│   • 文章很长很乱                 • 内容已经归纳好，有配图            │
│   • 可能被删了看不了            • 按行业分类打好了标签             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 两大核心目标

| 目标 | 说人话 | 解决什么问题 |
|------|--------|-------------|
| **后端：文章抓取存储** | 自动把公众号文章「抄写」下来存好 | 文章被删/过期看不到了 |
| **后端：打标签** | 给文章贴分类标签 | 找文章时能快速筛选 |
| **前端：定期推送** | 定时给客户发邮件 | 客户不用自己去找文章 |
| **前端：归纳总结** | 帮文章写摘要、配图 | 客户看一眼就知道讲什么 |

## 五、防封号策略（需要了解的背景）

### 5.1 为什么要防封号？

> 打个比方：就像去图书馆借书，如果一次性借太多或者太频繁，书库管理员会请你出去。

微信公众号有类似的限制机制，所以我们的「抓取机器人」需要：

| 策略 | 通俗解释 |
|------|---------|
| **批次控制** | 一次最多抓3个公众号，然后休息5分钟 |
| **随机延迟** | 每次抓取间隔30秒-2分钟（随机），模拟真人操作 |
| **每日限额** | 每天最多抓100次，避免触发限制 |
| **User-Agent 轮换** | 用不同的「身份」去抓取，降低被识别概率 |

### 5.2 防封号配置（由开发工程师维护）

| 参数 | 当前值 | 说明 |
|------|--------|------|
| 单批次公众号数 | 3 | 一次最多处理3个 |
| 批次间隔 | 5分钟 | 每批之间休息5分钟 |
| 请求延迟 | 30-120秒 | 随机等待时间 |
| 每日最大请求 | 100次 | 超额会暂停 |

⚠️ **注意**：如果抓取成功率突然下降，可能是触发了限制，需要调整参数。

## 七、数据存储策略

### 7.1 为什么要分层存储？

> 打个比方：就像冰箱里的食物，最近的要放冷藏层（随时取用），旧的要放冷冻层（长期保存）。

| 存储层级 | 数据年龄 | 存储方式 | 特点 |
|---------|---------|---------|------|
| **热数据** | 7天内 | 完整存储 + 缓存 | 访问快、占用稍多 |
| **温数据** | 7-30天 | gzip压缩存储 | 节省空间、访问稍慢 |
| **冷数据** | 30天以上 | 仅保存链接 | 最小空间、需要时再抓 |

### 7.2 存储预估

> ⚠️ **注意**：以下为开发参考，产品经理只需了解大概规模

| 订阅规模 | 文章数量 | 预估存储 |
|---------|---------|---------|
| 小型（当前） | 4个公众号 | ~10 MB |
| 中型 | 20个公众号 | ~50 MB |
| 大型 | 100个公众号 | ~250 MB |

## 十、测试要求（简化版）

### 10.1 测试类型

| 测试类型 | 说人话 | 目的 |
|---------|--------|------|
| **单元测试** | 单独检查每个零件 | 确保每个小功能正常 |
| **集成测试** | 把几个零件组装起来检查 | 确保零件之间配合正常 |
| **E2E测试** | 模拟用户操作走一遍全流程 | 确保整个系统能用 |

### 10.2 覆盖率要求

| 模块 | 最低要求 | 理想目标 |
|------|---------|---------|
| 核心业务逻辑 | 80% | 90% |
| 工具函数 | 90% | 95% |
| 整体项目 | 70% | 80% |

> 💡 **理解**：覆盖率就像考试题目的覆盖率，100% 不代表没有 bug，但覆盖率太低肯定有问题。
